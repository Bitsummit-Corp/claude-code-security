# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project shape

Open-source hardening reference for Anthropic's Claude Code CLI. pnpm workspace monorepo, TypeScript end-to-end, vitest for tests. Produces a `ccsec` CLI that compiles per-profile `~/.claude/settings.json` files plus a hook runtime.

The canonical repo is `https://github.com/Bitsummit-Corp/claude-code-governance`. `OWNERS.md` is the source of truth for project ownership; do not substitute a fork URL into docs even when the local `origin` remote points elsewhere.

## Big-picture architecture

### Compile pipeline

Both `ccsec compile` and `ccsec apply` call `compileProfile()` in `packages/cli/src/compiler.ts`:

1. Read `packages/settings/base.json`.
2. Read each overlay listed in `packages/settings/profiles/<profile>.json` `extends[]`.
3. Deep-merge top-level objects so overlays append to `permissions.deny`, `hooks.<event>`, etc.
4. Apply `profile.overrides`.
5. Resolve path tokens (`${HOME}`, `${SSH}`, `${AWS}`, `${TMP}`, `${KEYS}`) via `packages/core/src/path-tokens.ts`.
6. If `--target managed`, strip the `threat` field from deny entries.

`apply` then writes `~/.claude/settings.json` plus a sha256-stamped `.ccsec-lock.json` and (with `--rules`) installs the profile's `CLAUDE.md` template from `packages/rules/templates/`.

### Hook contract

Each hook is a directory under `archive/hooks-rc2/<name>/` with an `index.ts` exporting a manifest plus a `run(ctx)` function. Manifest fields: `name`, `event`, `matchers`, `threat`, `severity` (scalar or per-profile record), `profiles`, `timeout_ms`. `packages/core/src/runner.ts` is the in-process runner used by tests; the Claude Code harness does not call it in rc.2.

### Threat model and coverage

`docs/threat-model.md` is hand-curated; T-IDs are sequential and stable. `docs/coverage-matrix.md` and `docs/hooks/<name>.md` are auto-generated by `scripts/gen-coverage-matrix.mjs` and `scripts/gen-hook-docs.mjs`; regenerate via `pnpm gen:docs`. CI fails on stale generated docs. As of rc.2, 14 of 18 threats have hook coverage; T-007, T-009, T-014, T-016 are unmapped.

### Test layers

- **Unit:** `packages/<pkg>/tests/*.test.ts` and `archive/hooks-rc2/<name>/<name>.test.ts`.
- **Snapshot:** `packages/settings/snapshot.test.ts` compares each `compiled/<profile>.json` against the vitest snapshot.
- **Integration:** `tests/integration/*.test.ts` (in-process runner, not real Claude Code).
- **Bats:** `installers/macos/`.

There is no harness-contract test layer. Adding one is the structural fix for the rc.2 defects.

## Build and test

CI order is `lint -> build -> typecheck -> test`. Build comes before typecheck because typecheck depends on cross-package `dist/*.d.ts`. `--parallel` is omitted from `pnpm -r build` because the workspace has dep ordering (`core -> hooks -> cli`).

`pnpm test -- -u` updates vitest snapshots when settings change.

## Compiled artifacts

`packages/settings/compiled/{baseline,strict,regulated}.json` use placeholder home `/Users/x/`. Running the compile command from a developer machine resolves `${HOME}` to your real home directory and produces a noisy diff. Set `HOME=/Users/x` for the compile invocation, or hand-edit only the field that changed. The vitest snapshot at `packages/settings/__snapshots__/snapshot.test.ts.snap` must match the on-disk compiled JSON.

`pnpm build:settings` only regenerates `compiled/baseline.json`. Strict and regulated need explicit invocations of `node packages/cli/bin/ccsec.js compile --profile <p> --out packages/settings/compiled/<p>.json --target user --os macos --settings-root packages/settings`.

## Version surface

The project version lives in ten places: seven workspace `package.json` files, the `PKG_VERSION` constant in `packages/cli/src/index.ts`, the `VERSION` constant in `packages/cli/src/commands/apply.ts`, and the `ccsec_version` field in `packages/settings/base.json` (which propagates into the three compiled artifacts and the snapshot). `node packages/cli/bin/ccsec.js --version` should match the README badge.
